This is the fifth in a series of 5 G-assignments ("G" for "Godkendelse" and/or "Gruppeopgave") which you must pass in order to be admitted to the exam in OSM. To be more clear, you must pass 4/5 assignments to be admitted to the exam. Due to time-constraints, this assignment cannot be resubmitted. We encourage pair programming, so please form groups of 2-3 students.

This assignment is equally part practical, and part theoretical. Task 1 asks you to add filesystem management to the process abstraction in KUDOS. The remaining tasks are either purely theoretical, or unrelated to KUDOS.

1. Before you start

Along with this assignment, we hand out a fresh KUDOS source based on our reference solution to G2. Please use this as your baseline.

For background knowledge for this assignment, please consult the chapter on filesystems in the KUDOS documentation.

2. What to submit

Throughout this course, you should always submit entire, working source code tree in a single tar.gz archive. The uploaded archive should contain:

  1. A source code tree which includes your work for both tasks, and programs you used for testing.

  2. A short report where you document your code, discuss different ways to solve the tasks, explain the design decisions (why you preferred one particular way out of several choices), and assess your solution (especially through testing).

We assess the quality of your work be based on your report. Therefore, it is important to mention the most important parts of the C code that you wrote or modified. Pay particular attention to documenting changes to existing KUDOS code. You should also use comments in the C code itself (but use the report for long explanations).

The KUDOS code you hand in should compile without errors with the original setup. As KUDOS uses -Werror option in its Makefile, your code should also compile without warnings.

3. Task 1: System calls for the KUDOS filesystem

The system calls syscall_read and syscall_write were implemented in G1, but they only worked on stdin, stdout, and stderr. You will find our reference solution in the handed out KUDOS. Your task is to extend these system calls to work with general-purpose file handles. Other system calls are to be added, in order to open, close, create, remove (unlink), seek, and tell files.

Each process is to have its own, private list of open files. You will have to extend the KUDOS process abstraction to handle this. Files should be safely closed when a process terminates, and are otherwise non-transferable to children. You will find our reference implementation of the process abstraction in the handed out KUDOS.

Implement the system calls as mentioned below, following their description in the KUDOS documentation as closely as possible. In case of ambiguities, clearly state your assumptions in the comments and report. Try to identify all possible error and race conditions.

...
#define SYSCALL_OPEN      0x201
#define SYSCALL_CLOSE     0x202
#define SYSCALL_SEEK      0x203
#define SYSCALL_READ      0x204
#define SYSCALL_WRITE     0x205
#define SYSCALL_CREATE    0x206
#define SYSCALL_DELETE    0x207
#define SYSCALL_FILECOUNT 0x208
#define SYSCALL_FILE      0x209
...
#define FILEHANDLE_STDIN    0
#define FILEHANDLE_STDOUT   1
#define FILEHANDLE_STDERR   2
...

Use only calls to the virtual filesystem (VFS) layer of KUDOS, and the error codes defined there (see kudos/fs/vfs.[hc]). Do not use any parts of the trivial filesystem (TFS) directly. The use of kernel panic (rather than good use of locks and error return codes) should be your last resort.

4. Task 2: Filesystem Trivia

The block allocation table is a bitmap which records the free and reserved blocks on the disk, one bit per block, 0 meaning free and 1 reserved. For a 512-byte block size, the allocation table can hold 4096 bits, resulting in a 2MB disk. Note that the allocation table includes also the three first blocks, which are always reserved.

…​

The master directory consists of a single disk block, containing a table of the following 20-byte entries. This means that a disk with a 512-byte block size can have at most 25 files (512/20 = 25.6).

…​

With a 512-byte block size, the maximum size of a file is limited to 127 blocks (512/4 − 1) or 65024 bytes.

— KUDOS documentation
Trivial Filesystem

  1. Assuming a 512-byte block size, how many blocks do you need for the block allocation table to support disks of up to 4MB? 16MB? 512MB? 2GB?

  2. Can we fully utilize a 2MB TFS volume? If not, how many bytes are guaranteed to be wasted?

  3. What is the optimal number of blocks, i.e. such that as little space as possible is guaranteed to be wasted, to use for a TFS volume with a block-size of 512 bytes?

  4. Assume again a block size of 512 bytes. Consider now that TFS is modified such that the last word of a file header block is a pointer to a single indirect block. How big can the TFS files get, in KB?

  5. Now, consider modifying TFS again such that the next-to-last word of a file header block is a pointer to a single indirect block, and the last word is a pointer to a double-indirect block. How big can the files get, in MB?

6. Task 4: TLB Trivia

Read the Homework text (but not the questions!) at the end of OSTEP, Chapter 19, Paging: Faster Translations (TLBs). Consider the following implementation:

#include <stdio.h>

#define ASIZE 2000000
#define ITERATIONS 1000000000
#define PAGESIZE 4096

int main()
{
   int a[ASIZE];
   int jump = PAGESIZE/sizeof(int);
   printf("NUMPAGES: %d", NUMPAGES);
   for (int j = 0; j < ITERATIONS/NUMPAGES; j++)
      for (int i =0; i < NUMPAGES*jump; i += jump) {
         a[i] += 1;
      }
}

Compile and run this program with NUMPAGES set to 2, 4, 8, 16, 32, 64, 128, 256, 512, 1024, and time the execution using the time utility:

$ cc -DNUMPAGES=2 -o tlb tlb.c && time ./tlb
$ cc -DNUMPAGES=4 -o tlb tlb.c && time ./tlb
$ cc -DNUMPAGES=8 -o tlb tlb.c && time ./tlb
...

List the user times in a table for each NUMPAGES value. Explain your results.

7. Challenge of the week

"Challenge of the week" is a completely optional, probably tough, but enlightening part of the assignment. Good solutions will be called up during the final lecture, and awarded a student-friendly prize.

Virtual Memory promises large amounts of memory, often termed Random Access Memory. But the implementation of Virtual Memory makes assumptions about program as to make the implementation of Virtual Memory efficient, e.g., locality of reference. Indeed, if we access virtual memory in a more random manner, we usually experience a performance degradation. In the assignment above, we shed some light on the performance impact of TLBs.

This challenge is to investigate the effects of Virtual Memory implementation in general.

Write a program that has a two-dimensional array A[1024][1024]. Write two versions of the program: Both add one to each of the elements in the array, but the first one does it in row order, while the other in column order.

Loop a number of times, each iteration adding one to each array element as described above, so that the running time of the program is, say 10 seconds for the row order program. Then time both programs. Show the results. Try explaining the difference in timings.

Try on as many different machines that you can lay your hands on (or access over the internet).

IF you are adventurous, we can give you access to a plethora of machines on the internet testbed platform called Planet-Lab. The testbed has more than 1,000 individual machines distributed across the world. Details at: https://www.planet-lab.eu. Sign up at: https://www.planet-lab.eu/db/persons/register.php — IMPORTANT pick University of Copenhagen, DIKU as the site.
