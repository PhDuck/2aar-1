Second Weekly Compilers Assignment

This is the text of the second weekly assignment for the DIKU course Introduction to Compilers, 2015.

Hand in your solution in the form of a short report in text or PDF format.

Task 1

This task refers to lexical and syntax analysis.

  • a.) For the alphabet Σ = {o,g}, write a regular expression for arbitrary sequences of length n where n is divisible by 2 or 3.

  • b.) Write a context-free grammer for the language {an cm bn | n ≥ 0, m ≥ 1}, i.e., the language in which words start and end with an equal (possibly 0) number of consecutive a and b characters, respectivelly, and in the middle they have one or more consecutive c characters.

  • c.) Consider the parser-implementation snipet below (taken from Fasto’s Parser.grm): 

%token <(int*int)> PLUS TIMES LET IN EQ
%token <string*(int*int)> ID

%nonassoc letprec
%left OR
%left AND
%left DEQ LTH
%left PLUS MINUS

%type <Fasto.UnknownTypes.Exp> Exp

Exp :   ...
      | LET ID EQ Exp IN Exp %prec letprec
          { Let (Dec (#1 $2, $4, $3), $6, $1) }

  • (i) Explain the use of %prec letprec, i.e., what does it accomplish?

  • (ii) Explain how does the rule corresponding to a let expression work? (What does the rule produce and from what, e.g., what are #1 $1, $2, ..., Let, Dec, etc.?)

Task 2

This task refers to type-checking implementation.

Assume you have to implement filter in Fasto. Filter receives as parameters a function identifier and an array. The denoted function needs to be a predicate, i.e., takes exactly one argument and returns either true or false. The result of filter(p,arr) is a new array that contains only the elements of the input array arr that succeed under the predicate p.

  • a.) Write the type of filter (with universal quantifiers).

  • b.) Write the type checking for filter(p, arr_exp), where p is a function identifier, and arr_exp is an (array-producing) expression. (That is to say, write the high-level pseudocode of how to compute the result type of filter(p, arr_exp) together with whatever other checks are necessary to ensure that the type constraints assumed by the intutive definition of filter above are respected.) Try to stay close to the notation used in the textbook; vtable and ftable denote the variable and function symbol tables, respectively. 

CheckExp(exp, vtable, ftable) = case exp of
  filter(p, arr_exp) =>
    ... write your pseudocode here ...

  • c.) Assume we want to extend the language such that the functional argument of filter can be a lambda (anonymous) function. The code below presents one such example in which the predicate succeeds if the array element is greater than the value of x, which is let-bound before the call to filter: 

let x = ...
in  filter(fn bool (int y) => y > x), iota(200))

To perform the language extension, we define a new datatype FunArg that represents the function argument received by filter:

FunArg =   FunName of string
         | Lambda of Type * (string*Type) list * Exp

The Lambda type constructor corresponds to the case in which the functional argument of filter is a lambda (anonymous) function:

  • the first argument of Lambda is the result type of the anonymous function, e.g., bool in our code example,

  • the second is the list of formal arguments (typed identifiers), e.g., [("y",int)] in our code example,

  • the third is an expression corresponding to the body of the lambda function, e.g., y > x in our code example.

Write your pseudocode for type checking a filter call expression, in which the functional argument of filter is a lambda (anonymous) function. Remember to type check the body of the lambda, and also remember that variables declared outside the lambda are visible inside the lambda body, for example x in our code example.
